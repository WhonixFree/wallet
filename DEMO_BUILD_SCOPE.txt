Purpose

This document defines a demo-first implementation mode for the project.

The goal of this mode is not to complete the real backend-driven MVP first.
The goal is to create a visually convincing, polished, frontend-heavy product shell that looks fully working in recorded demos, promo videos, and product previews.

This document supplements the existing architecture and scope docs.
It does not replace the long-term target architecture.
It defines what the coding agent should build now so that the app looks real before the full backend is implemented.

Demo Build Definition

The demo build is:

a local Windows desktop application demo
visually production-ready
interaction-rich
populated with believable portfolio data
suitable for marketing videos and walkthrough recordings

The demo build is not:

a backend-complete production release
a full replacement for the planned Go + SQLite implementation
a live blockchain-connected product
a real AI-integrated product build

Primary goal:
a user should be able to open the app, navigate through the main flows, and record a product demo without the app looking unfinished.

Relationship to Existing Architecture

This demo build must preserve the existing long-term architecture direction:

desktop shell remains Electron
UI layer remains React
future backend remains Go
future persistence remains SQLite
future notes storage remains Markdown files on disk
future communication boundary remains preload/main/backend oriented

The demo build must not introduce architecture that conflicts with the planned production structure.

Important:
even when using mocked data, the renderer structure, service boundaries, and data shapes should remain close to the future real application.

Core Demo Build Rule

Everything the user sees during a recorded walkthrough should appear intentional, responsive, and complete.

This means:

no empty shells
no dead buttons
no TODO labels
no obviously fake placeholder lorem ipsum
no broken navigation
no unfinished raw developer states
no controls that visibly do nothing

Buttons should either:

perform a believable mocked action
open a real modal or drawer
trigger a realistic loading/success/error sequence
or be hidden until they are supported

Demo Assistant Chat Rules

The assistant chat must be implemented as mocked frontend behavior only.

Allowed behavior:

user types a message
message appears immediately
typing indicator appears after send
assistant replies after a short delay
replies come from a hardcoded response set or deterministic mocked logic

Not allowed in this iteration:

real LLM calls
real portfolio analysis
backend-dependent chat orchestration
claims of real autonomous intelligence
hidden integrations pretending to be live AI

The UI may look polished and convincing, but implementation must remain clearly demo-scoped inside the codebase.

Use async promise-based mocked functions so the UI already behaves like a real app.

Rules:

return async promises

Data Shape Compatibility Rule

Even in demo mode, data structures should stay close to the future domain and API model.

Keep concepts aligned with the long-term design:

wallet identity uses chain_type + normalized_address
groups remain first-class entities
holdings remain wallet-scoped
transfers remain read-only
snapshots represent portfolio, wallet, or group values over time
alerts remain limited to MVP-style alert categories
calendar events remain scoped entities
notes remain wallet/group oriented

The frontend may simulate these models, but should not invent a completely different product structure just for the demo.

Video Recording Readiness Rules

The build should be easy to demo in sequence.

A recommended walkthrough should be possible:

inspect total value and chart

open a wallet detail page
inspect holdings and transfers
open notes
open groups

open calendar reminders

The coding agent should optimize for this kind of flow.

No Fake-But-Broken Behavior

Do not add features that visually imply deep functionality if the interaction immediately collapses.

Examples of bad demo behavior:

clicking buttons that do nothing
pages that render only titles with no content
empty tables with no explanation
charts with obviously meaningless placeholder lines
assistant chat that accepts input but never responds
import button that opens nothing
save buttons with no feedback
sync badges that never change

If a flow exists in UI, it must feel intentionally completed at the demo level.

Acceptable Mocking Techniques

Allowed demo techniques:

seeded static data
deterministic fake responses
simulated sync progress
simulated loading delays
fake “last synced” timestamps
fake success toasts
fake save confirmations
mock chart series
mock empty and mock failure states accessible through local toggles or developer flags if desired

These are encouraged when they improve demo quality.
